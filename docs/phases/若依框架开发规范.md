## 一、参数校验规范

### 手动校验 → 注解校验对照表

| 手动校验写法 | 推荐替换方式 | 放在哪里 | 说明 |
|---|---|---|---|
| `if (obj == null) throw ...` | `@Valid @RequestBody XxxDto dto` | Controller 入参 | 整个请求体为空时交给框架校验 |
| `if (id == null) throw ...` | `@NotNull(message = "ID不能为空")` | DTO 字段 | 适合 `Long id`、`Integer id` |
| `if (name == null && "".equals(name)) throw ...` | `@NotBlank(message = "名称不能为空")` | DTO 字段 | 适合字符串，不能为 `null`、空串、纯空格 |
| `if (list == null && list.isEmpty()) throw ...` | `@NotEmpty(message = "列表不能为空")` | DTO 字段 | 适合 `List`、`Set`、数组、字符串 |
| `if (str.length() > 64) throw ...` | `@Size(max = 64, message = "长度不能超过64个字符")` | DTO 字段 | 适合字符串、集合长度限制 |
| `if (str.length() < 2 && str.length() > 20) throw ...` | `@Size(min = 2, max = 20, message = "长度必须在2到20之间")` | DTO 字段 | 常用于名称、账号 |
| `if (age < 0) throw ...` | `@Min(value = 0, message = "年龄不能小于0")` | DTO 字段 | 数字最小值 |
| `if (num > 100) throw ...` | `@Max(value = 100, message = "数量不能大于100")` | DTO 字段 | 数字最大值 |
| `if (amount <= 0) throw ...` | `@DecimalMin(value = "0", inclusive = false, message = "金额必须大于0")` | DTO 字段 | `BigDecimal` 金额校验 |
| `if (!phone.matches(...)) throw ...` | `@Pattern(regexp = "...", message = "手机号格式不正确")` | DTO 字段 | 手机号、编码、身份证号等格式校验 |
| `if (!email.matches(...)) throw ...` | `@Email(message = "邮箱格式不正确")` | DTO 字段 | 邮箱校验 |
| `if (startDate == null) throw ...` | `@NotNull(message = "开始时间不能为空")` | DTO 字段 | 日期不能为空 |
| `if (date 是未来时间) throw ...` | `@Past(message = "时间必须早于当前时间")` | DTO 字段 | 生日、历史时间 |
| `if (date 是过去时间) throw ...` | `@Future(message = "时间必须晚于当前时间")` | DTO 字段 | 预约时间、截止时间 |
| `if (!Arrays.asList("0","1").contains(status)) throw ...` | `@Pattern(regexp = "01", message = "状态值不合法")` | DTO 字段 | 状态、枚举值校验 |
| `if (!"NOT_DELETE".equals(deleteFlag)) throw ...` | 通常不要让前端传 `deleteFlag` | 后端自动赋值 | 删除标识一般不应该作为新增入参 |
| `if (pageNum == null && pageSize == null) throw ...` | 使用 RuoYi `startPage()` | — | 分页参数由框架处理 |
| `if (userId == null)` 出现在 ServiceImpl | `@NotNull` 放到请求 DTO | DTO + Controller | Service 不处理基础入参非空 |
| `if (StringUtils.isBlank(name))` 出现在 ServiceImpl | `@NotBlank` 放到 DTO | DTO + Controller | Service 只保留业务规则 |
| `if (dto.getXxx() == null)` 多个字段连续判断 | 多个字段分别加校验注解 | DTO 字段 | 让框架统一返回参数错误 |
| `if (personUser == null)` | `@Valid @RequestBody CmPersonUserBindDto dto` | Controller 入参 | 请求体校验放 Controller |

## 二、Java 开发规范

### 2.1 命名规范

- **类名**：使用 `UpperCamelCase`，如 `UserController`、`OrderServiceImpl`
- **方法名**：使用 `lowerCamelCase`，动词开头，如 `createUser()`、`calculateTotalPrice()`
- **变量名**：使用 `lowerCamelCase`，如 `userName`、`orderList`
- **常量名**：使用 `UPPER_SNAKE_CASE`，如 `MAX_RETRY_COUNT = 3`、`DEFAULT_TIMEOUT = 5000`
- **包名**：全小写，使用公司域名反转 + 模块名，如 `com.aiplatform.user.service`
- **接口名**：使用 `UpperCamelCase`，前缀 `I`（如 `IUserService`），团队需统一
- **枚举类**：类名使用 `UpperCamelCase`，枚举值使用 `UPPER_SNAKE_CASE`

```java
enum OrderStatus { CREATED, PAID, SHIPPED }
```

### 2.2 代码结构

- **类成员顺序**：静态变量 → 成员变量 → 构造方法 → 静态方法 → 成员方法
- **方法参数**：不超过 5 个，超过时使用对象封装
- **方法长度**：单个方法代码不超过 100 行，复杂逻辑需拆分
- **类长度**：单个类代码不超过 1000 行，过大时按职责拆分

### 2.3 语法规范

#### 异常处理

- 必须捕获具体异常，禁止 `catch (Exception e)`
- 异常信息需包含具体上下文

```java
throw new IllegalArgumentException("用户 ID 不能为空: " + userId);
```

- 禁止空 `catch` 块，至少记录日志

#### 集合处理

- 初始化指定容量：`List<User> users = new ArrayList<>(10)`
- 遍历使用增强 for 循环：`for (User user : users) {}`
- 集合判空使用 `CollectionUtils.isEmpty(collection)`

#### 注释规范

**类注释**（包含功能描述、作者、创建日期）：

```java
/**
 * 用户服务实现类，处理用户 CRUD 操作
 *
 * @author 张三
 * @date 2025-08-24
 */
```

**方法注释**（包含功能、参数、返回值、异常）：

```java
/**
 * 根据 ID 查询用户
 *
 * @param userId 用户 ID
 * @return 用户信息，不存在返回 null
 * @throws IllegalArgumentException 当 userId 为 null 时抛出
 */
```

### 2.4 性能与安全

- **字符串拼接**：循环中使用 `StringBuilder`，避免 `+` 运算符
- **资源释放**：IO 流、数据库连接等必须在 `try-with-resources` 中使用
- **线程安全**：共享变量需加锁或使用线程安全容器
- **避免魔法值**：使用常量或枚举替代硬编码数值
- **安全校验**：所有外部输入必须进行校验，防止 XSS、SQL 注入

### 2.5 框架规范

- 服务层命名后缀为 `Service`，实现类后缀为 `ServiceImpl`
- 控制器命名后缀为 `Controller`
- 事务注解 `@Transactional` 加在业务层，指定 `rollbackFor`

## 三、Python 开发规范

### 3.1 命名规范

- **函数名、变量名**：使用 `snake_case`，如 `user_name`、`calculate_price()`
- **类名**：使用 `UpperCamelCase`，如 `UserManager`
- **常量**：使用 `UPPER_SNAKE_CASE`，如 `MAX_RETRY = 3`
- **模块名**：全小写，简短，如 `user_service.py`

### 3.2 代码格式

- **缩进**：严格使用 4 个空格
- **空行**：
  - 类定义后空两行
  - 函数定义后空一行
  - 逻辑块之间空一行
- **行长度**：不超过 120 个字符，过长需换行

```python
# 推荐
result = long_function_name(var_one, var_two,
                            var_three, var_four)
```

### 3.3 导入规范

- **导入顺序**：标准库 → 第三方库 → 本地应用/库
- **禁止使用** `from module import *`
- **分组导入**：

```python
# 标准库
import os
import sys

# 第三方库
import requests
from flask import Flask

# 本地库
from .models import User
```

### 3.4 注释规范

**文档字符串**：使用三双引号，包含功能、参数、返回值（注意与 Java 的区别）

```python
def calculate_area(radius):
    """计算圆的面积

    Args:
        radius (float): 圆的半径

    Returns:
        float: 圆的面积

    Raises:
        ValueError: 当半径为负数时抛出
    """
```

**行内注释**：与代码间隔两个空格

```python
total = price * quantity  # 计算总价
```

### 3.5 语法规范

- **条件判断**：避免单行 if 语句
- **列表推导**：简单逻辑可用，复杂逻辑应拆分
- **异常处理**：明确捕获特定异常，不使用 `except Exception`
- **类型提示**：关键函数添加类型提示

```python
def add(a: int, b: int) -> int:
    return a + b
```

## 四、SQL 开发规范

> SQL 语句可读性较差，特别是当 SQL 行数超过二三十行时，调试也比 Java、Python 等语言困难。SQL 规范建立在对数据库设计了解的基础上，数据库设计文档对新人尤其必要。

### 4.1 命名规范

| 类别 | 规范 | 示例 |
|---|---|---|
| 表名 | `snake_case`，复数形式，禁止缩写 | `users`、`order_items`、`product_categories` |
| 列名 | `snake_case`，避免 SQL 关键字 | `user_name`、`order_amount`、`created_at` |
| 主键 | 单表主键统一 `id` | `id` |
| 联合主键 | `{关联表名}_id` | `user_id`、`role_id` |
| 外键 | `{关联表名}_id` | `user_id`、`parent_id`（自关联） |
| 索引 | 前缀 `idx_` + 表名 + 列名 | `idx_users_email`、`idx_orders_user_id_created_at` |
| 视图 | 前缀 `v_` + 功能描述 | `v_active_users`、`v_order_statistics` |
| 存储过程 | 前缀 `sp_` + 动作 + 功能 | `sp_update_user_status`、`sp_generate_order_no` |
| 函数 | 前缀 `fn_` + 功能 | `fn_calculate_tax`、`fn_format_date` |

### 4.2 格式规范

- **关键字**：全部大写（`SELECT`、`FROM`、`WHERE`、`JOIN`、`GROUP BY`）
- **缩进与换行**：子句单独换行，子句内字段/条件缩进 4 个空格
- **逗号位置**：字段后加逗号，放在行尾（而非行首），便于新增/删除字段

```sql
-- 推荐
SELECT
    user_id,
    user_name,
    email,
    created_at
FROM
    users
WHERE
    status = 1                              -- 1 表示正常状态
    AND created_at BETWEEN '2025-01-01' AND '2025-08-24'
ORDER BY
    created_at DESC
LIMIT 10;
```

**注释格式**：

- 单行注释：使用 `-- 注释内容`
- 块注释：使用 `/* 多行注释内容 */`
- 复杂查询必须添加注释，说明业务场景（如 `-- 统计近 30 天各部门新增用户数`）

### 4.3 表设计规范

#### 必含字段

每张业务表必须包含 4 个基础字段，确保数据可追溯、可管理：

```sql
CREATE TABLE users (
    id          BIGINT        PRIMARY KEY AUTO_INCREMENT  COMMENT '用户主键 ID，自增',
    create_date TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP  COMMENT '创建时间，自动填充当前时间',
    update_date TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP  COMMENT '更新时间，修改时自动更新',
    is_delete   TINYINT       NOT NULL DEFAULT 0          COMMENT '逻辑删除标识：0-未删除，1-已删除'
    -- 其他业务字段...
) COMMENT = '用户基础信息表';
```

#### 字段类型选择

- **整数类型**：按数据范围选择最小合适类型（`TINYINT` 1B / `SMALLINT` 2B / `INT` 4B / `BIGINT` 8B）
- **字符串类型**：明确长度，`VARCHAR` 用于变长字符串，`CHAR` 用于定长字符串，禁止滥用 `VARCHAR(255)`
- **时间类型**：使用 `TIMESTAMP`（自动适配时区），禁止使用 `DATETIME`

#### 约束设置

- 非必要字段不允许为 `NULL`，需设置默认值（如 `status` 默认 1、`remark` 默认 `''`）
- 唯一约束：重复数据需加 `UNIQUE`（如 `email`、`phone`）
- 外键约束：关联表需加外键（若用 ORM 框架可在应用层维护）

### 4.4 查询规范

#### 禁止 `SELECT *`

```sql
-- 推荐
SELECT id, user_name, email FROM users WHERE is_deleted = 0;

-- 不推荐
SELECT * FROM users WHERE is_deleted = 0;
```

#### WHERE 条件优化

- 优先使用索引字段过滤（如 `id`、`user_id`、`created_at`）
- 避免在索引字段上使用函数（如 `DATE(created_at) = '2025-08-24'` 改为 `BETWEEN`）
- 等值条件（`=`）放在前面，范围条件（`BETWEEN`/`IN`）放在后面

#### JOIN 操作规范

```sql
-- 推荐
SELECT
    u.user_name,
    o.order_no,
    o.order_amount
FROM
    users u                             -- 小表作为驱动表
    INNER JOIN orders o ON u.id = o.user_id  -- o.user_id 为索引字段
WHERE
    u.is_deleted = 0
    AND o.is_deleted = 0;
```

- 明确指定 JOIN 类型（`INNER JOIN`/`LEFT JOIN`/`RIGHT JOIN`）
- 小表驱动大表，减少循环次数
- JOIN 条件必须包含索引字段

#### 排序与分页

```sql
-- 优化前（偏移量 10000，需扫描 10100 行）
SELECT * FROM orders WHERE is_deleted = 0 LIMIT 10000, 10;

-- 优化后（仅扫描 10 行，需 id 连续）
SELECT * FROM orders WHERE id > 10000 AND is_deleted = 0 LIMIT 10;
```

- 大表排序（> 10 万行）需添加排序字段索引，避免 `Using filesort`
- 大偏移量（`OFFSET > 1000`）避免使用 `LIMIT offset, size`，改用 ID 过滤

#### 子查询 → JOIN 转换

复杂子查询改为 JOIN 操作，提升性能和可维护性：

```sql
-- 子查询（不推荐，多层嵌套难维护）
SELECT user_name FROM users WHERE id IN (SELECT user_id FROM orders WHERE order_amount > 1000);

-- JOIN 改写（推荐）
SELECT DISTINCT u.user_name
FROM users u
    INNER JOIN orders o ON u.id = o.user_id
WHERE o.order_amount > 1000 AND u.is_deleted = 0;
```

### 4.5 写入与更新规范

#### INSERT 操作

```sql
-- 推荐（批量插入）
INSERT INTO users (user_name, email, phone) VALUES
    ('张三', 'zhangsan@test.com', '13800138000'),
    ('李四', 'lisi@test.com', '13900139000'),
    ('王五', 'wangwu@test.com', '13700137000');

-- 不推荐（单条插入，执行效率低）
INSERT INTO users (user_name, email, phone) VALUES ('张三', 'zhangsan@test.com', '13800138000');
```

- 明确指定字段名，不依赖表结构顺序
- 批量插入使用 `VALUES` 多组值，单次不超过 1000 条

#### UPDATE 操作

```sql
-- 推荐（指定 WHERE 条件）
UPDATE users SET user_name = '张三三', updated_at = NOW() WHERE id = 123 AND is_deleted = 0;

-- 危险（无 WHERE 条件，全表更新！）
UPDATE users SET user_name = '张三三';
```

- 必须包含 `WHERE` 条件，禁止全表更新
- 避免更新主键字段，禁止更新 `created_at`

#### 删除操作

```sql
-- 推荐（逻辑删除）
UPDATE users SET is_deleted = 1, updated_at = NOW() WHERE id = 123;

-- 谨慎使用（物理删除，数据不可恢复）
DELETE FROM users WHERE id = 123;
```

- 优先使用逻辑删除（更新 `is_deleted` 字段），保留数据追溯能力
- 物理删除需经审批（如日志表清理）

#### 事务控制

```sql
BEGIN;  -- 开启事务

-- 1. 插入订单数据
INSERT INTO orders (order_no, user_id, order_amount) VALUES ('ORD20250824001', 123, 999.00);

-- 2. 扣减商品库存（需判断库存是否充足）
UPDATE products SET stock = stock - 1 WHERE id = 456 AND stock >= 1;

-- 3. 检查库存扣减结果，失败则回滚
IF ROW_COUNT() = 0 THEN
    ROLLBACK;
    SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '商品库存不足';
END IF;

COMMIT;  -- 提交事务
```

- 多表写入/更新必须使用事务，保证原子性

### 4.6 安全规范

#### 防止 SQL 注入

必须使用参数化查询，禁止字符串拼接 SQL：

```java
// Java（PreparedStatement）
String sql = "SELECT * FROM users WHERE user_name = ? AND is_deleted = 0";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, userName);   // 参数化传值，避免注入
```

```python
# Python（MySQLdb）
cursor.execute(
    "SELECT * FROM users WHERE user_name = %s AND is_deleted = 0",
    (userName,)   # 元组传参，禁止字符串拼接
)
```

#### 敏感数据保护

- 密码存储：使用 BCrypt、SHA256 等算法加密，禁止明文存储
- 敏感字段查询：禁止在日志中输出手机号、邮箱等敏感数据

#### 权限控制

- 按最小权限原则分配数据库账号
- 查询账号仅授予 `SELECT` 权限，读写账号授予 `SELECT/INSERT/UPDATE` 权限
- 禁止授予 `DROP/ALTER/TRUNCATE` 等高危权限
- 禁止使用 `root` 账号直接连接应用

### 4.7 性能优化规范

#### 索引优化

- 高频查询字段（`WHERE`/`JOIN`/`ORDER BY`）添加索引
- 避免过度索引：单表索引不超过 5 个（索引会降低写入性能）
- 长字符串可指定前缀索引：`idx_users_remark(50)`

#### SQL 执行优化

- 避免 `DISTINCT`（可通过 `GROUP BY` 替代，或优化 JOIN 逻辑减少重复）
- 禁止 `NOT IN`，改用 `LEFT JOIN + IS NULL`：

```sql
-- 不推荐（NOT IN 可能导致全表扫描）
SELECT * FROM users WHERE id NOT IN (SELECT user_id FROM orders);

-- 推荐（LEFT JOIN + IS NULL）
SELECT u.*
FROM users u
    LEFT JOIN orders o ON u.id = o.user_id
WHERE o.user_id IS NULL AND u.is_deleted = 0;
```

#### 大表优化

- 数据量超过 1000 万的表需分区（按时间/范围）
- 历史数据归档：将超过 1 年的冷数据归档到归档表，减少主表数据量

## 五、RuoYi 框架专项规范

### 5.1 项目目录结构

```
backend/
├── ruoyi-admin/              # 启动模块，Controller 层
├── ruoyi-common/             # 公共模块（domain、utils、constants、enums、exception）
├── ruoyi-framework/          # 框架核心（Security、AOP、filter、interceptor）
├── ruoyi-system/             # 系统管理业务（用户、角色、菜单、部门）
├── ruoyi-generator/          # 代码生成器
└── ruoyi-quartz/             # 定时任务
```

### 5.2 分层架构规范

```
Controller  →  接收参数、调用 Service、返回 AjaxResult
   ↓
Service     →  业务逻辑、事务控制、调用 Mapper
   ↓
Mapper      →  数据访问（MyBatis XML / 注解）
   ↓
Domain      →  实体类，对应数据库表
```

**硬性规则**：

- **Controller 层**：只做参数校验和结果返回，不写业务逻辑
- **Service 层**：处理全部业务逻辑，事务注解 `@Transactional` 加在此层
- **Mapper 层**：只做数据访问，不写业务逻辑
- **禁止 Service 之间循环依赖**，必要时抽取公共模块
- **禁止 Controller 直接调用 Mapper**，必须经过 Service

### 5.3 实体类规范

- 数据库表对应实体类放在 `domain` 包下
- 实体类需继承 `BaseEntity`，自动获得 `createTime`、`updateTime`、`createBy`、`updateBy`、`remark` 等公共字段
- 查询条件封装的 DTO 放在 `domain` 包，命名后缀为 `QueryDto`

### 5.4 统一响应格式

```java

@GetMapping("/list")
public TableDataInfo list(SysUser user, HttpServletRequest request) {
```

- `AjaxResult.success()` / `AjaxResult.error()` 统一返回格式
- 分页查询返回 `TableDataInfo`（调用 `getDataTable(list)`）
- 新增/修改/删除返回 `AjaxResult`（调用 `toAjax(rows)`）

### 5.5 分页规范

使用 RuoYi 封装的 `startPage()` 方法：

```java
@GetMapping("/list")
public TableDataInfo list(SysUser user) {
    startPage();                                    // 自动解析 pageNum、pageSize
    List<SysUser> list = userService.selectUserList(user);
    return getDataTable(list);
}
```

- **必须在 Mapper 查询之前调用** `startPage()`
- `startPage()` 会自动从请求中获取 `pageNum` 和 `pageSize` 参数
- 分页原理基于 PageHelper，`startPage()` 后的第一条 SQL 会自动分页
- **禁止在 `startPage()` 和查询之间插入其他 SQL 操作**

### 5.13 代码生成器使用规范

1. 先在数据库中创建业务表（必须包含 `create_time`、`update_time`、`create_by`、`update_by`、`remark` 字段）
2. 登录系统 → 系统工具 → 代码生成 → 导入表
3. 编辑生成配置：设置生成模板、包路径、模块名、业务名、作者
4. 生成代码并下载
5. 将后端代码（`domain`、`mapper`、`service`、`controller`）放入对应模块
6. 将前端代码放入 `frontend/src/views/` 对应目录
7. 在 SQL 菜单中手动添加菜单路径即可

### 5.14 异常处理规范

```java
// 业务异常使用 ServiceException（会被全局异常处理器捕获）
throw new ServiceException("用户账号已存在");

// 系统内部错误使用 RuntimeException
throw new RuntimeException("系统内部错误");
```

- `ServiceException` 抛出的消息会直接返回给前端，常用于业务校验失败
- 避免在 `catch` 块中吞掉异常，至少要记录日志
- 不要将详细的内部错误信息暴露给前端（安全性）

### 5.15 Redis 缓存规范

- 登录 Token 存储在 Redis，key 格式：`login_tokens:{token-uuid}`
- 用户权限信息缓存 key 格式：`login_tokens:{token-uuid}`（同一个 key，value 中包含 `LoginUser` 对象）
- 验证码缓存 key 格式：`captcha_codes:{uuid}`
- 系统参数缓存 key 格式：`sys_config:sys.index.skinName`
- **禁止直接在业务代码中操作 Redis**，应通过框架提供的 `TokenService`、`ConfigService` 等封装类访问

### 5.16 定时任务规范

- 定时任务类放在 `ruoyi-quartz` 模块
- 任务类需实现 `Job` 接口或继承 `AbstractQuartzJobBean`
- 支持 `allowConcurrent` 控制是否并发执行
- 任务参数通过 `invokeTarget` 配置（格式：`beanName.methodName(params)`）
- 禁止在定时任务中执行耗时操作（> 30s），超时任务需拆分或异步化

### 5.17 安全规范

- **密码存储**：必须使用 BCrypt 加密，调用 `SecurityUtils.encryptPassword()`
- **Token 管理**：JWT Token 过期时间默认 30 分钟，可通过 `application.yml` 配置
- **XS S 防护**：全局过滤器已内置 XSS 过滤
- **文件上传**：必须校验文件类型和大小，禁止上传可执行文件
- **API 接口**：所有增删改操作接口必须有 `@PreAuthorize` 权限注解
- **敏感配置**：数据库密码、Redis 密码等使用配置文件管理，不硬编码
